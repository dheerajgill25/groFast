## Test cases this task must make pass

### TC-SEC-004 — OWASP ASVS L2 spot checks: injection, IDOR, XSS
- **Type:** security
- **Requirement:** NFR-5
- **Preconditions:** Staging; two customer accounts A and B
- **Steps:** 1. SQLi payloads on search/order-lookup/CSV imports 2. Fetch B's order/address/invoice with A's token (IDOR) 3. Stored-XSS payloads in SKU name/address/rating fields rendered in admin
- **Expected:** No injection (parameterized everywhere); all cross-user object access 403/404; payloads rendered inert (escaped) in every surface; ASVS L2 checklist items ticked in CI evidence
- **Test data:** `payload set `sec/asvs_l2.md``
- **Priority:** P0

### TC-SEC-005 — No raw card/UPI instrument data anywhere (PCI SAQ-A)
- **Type:** security
- **Requirement:** NFR-5, NFR-6
- **Preconditions:** Full E2E payment executed with tracing + debug logs on
- **Steps:** 1. Complete card + UPI sandbox payments 2. Sweep DB dumps, logs, traces, Redis, object storage for PAN/instrument patterns
- **Expected:** Zero raw instrument data at rest or in logs (regex sweep clean); only gateway tokens/refs stored; checkout served from gateway-hosted fields/SDK
- **Test data:** `PAN regex sweep `sec/pan_sweep.sh``
- **Priority:** P0

### TC-SEC-007 — DPDP consent captured, versioned, and enforced
- **Type:** security
- **Requirement:** NFR-6
- **Preconditions:** New signup flow
- **Steps:** 1. Sign up, inspect consent record 2. Bump consent version, log in as pre-bump user
- **Expected:** Versioned consent stored on user with timestamp + purpose; re-consent requested on version bump; processing gates respect consent flags
- **Priority:** P0

### TC-SEC-008 — DPDP deletion anonymizes PII, retains statutory records
- **Type:** security
- **Requirement:** NFR-6
- **Preconditions:** User with orders, payments, addresses, ratings
- **Steps:** 1. Submit deletion request 2. Complete workflow 3. Inspect DB + warehouse + logs
- **Expected:** Phone/name/address tombstone-hashed in place; user cannot log in; orders/payments/invoices/audit retained per GST/statutory carve-out; deletion itself audit-logged; confirmation issued to user
- **Priority:** P0

### TC-SEC-009 — Audit completeness matrix: every money/stock mutation audited
- **Type:** security
- **Requirement:** NFR-10, FR-22
- **Preconditions:** Staging with all flows executable
- **Steps:** 1. Execute one of each: GRN, adjustment, cycle count, reserve, release, price change, coupon apply, payment capture, refund, COD collection [COND-COD §13], commission record 2. Query `audit_event`
- **Expected:** 100%: each mutation has exactly one append-only audit row with actor, actor_role, before/after, reason; UPDATE/DELETE on `audit_event` blocked at DB level
- **Test data:** `mutation checklist `sec/audit_matrix.md``
- **Priority:** P0

### TC-SEC-010 — Client-tampered totals rejected — server recomputes money
- **Type:** security
- **Requirement:** NFR-5, FR-9
- **Preconditions:** Intercepting proxy on checkout
- **Steps:** 1. Modify bill total/discount/fee fields in checkout + payment requests
- **Expected:** Server recomputes bill from SKU prices + coupon rules; tampered values ignored/rejected; gateway amount always server-derived; attempt logged
- **Priority:** P0

### TC-SEC-011 — PII encrypted at rest and redacted in logs
- **Type:** security
- **Requirement:** NFR-5
- **Preconditions:** DB volume + column crypto configured
- **Steps:** 1. Inspect raw column bytes for phone/address 2. Look up user via HMAC phone index 3. Grep logs/traces for seeded phone number
- **Expected:** Columns ciphertext at rest; deterministic HMAC index resolves lookups without plaintext; zero PII hits in logs/traces (logger-layer redaction)
- **Test data:** `seeded phone `+919800000099``
- **Priority:** P1

### TC-SEC-001 — OTP brute-force and SMS-pump protections hold
- **Type:** security
- **Requirement:** NFR-5, FR-1
- **Preconditions:** Rate caps configured (3/min, 10/day per number; per-IP caps)
- **Steps:** 1. Script 10 OTP requests/min for one number 2. Rotate numbers from one IP 3. Brute-force 6-digit codes
- **Expected:** Per-number and per-IP caps enforced at edge + app; OTP compare constant-time; lockout/backoff after cap; alerts on pump pattern
- **Test data:** `attack scripts `sec/otp_abuse.ts``
- **Priority:** P0

### TC-SEC-002 — JWT expiry + refresh rotation + reuse detection
- **Type:** security
- **Requirement:** NFR-5
- **Preconditions:** Valid session
- **Steps:** 1. Use access token past 15 min 2. Rotate refresh 3. Replay the OLD refresh token
- **Expected:** Expired JWT rejected; rotation issues new pair; old-refresh reuse revokes the entire token family and forces re-login
- **Priority:** P0

### TC-SEC-003 — Audience claims isolate rider/customer/admin tokens
- **Type:** security
- **Requirement:** NFR-5, FR-29
- **Preconditions:** Valid rider JWT
- **Steps:** 1. Call customer order API and admin API with rider token 2. Vice versa with customer token
- **Expected:** All cross-audience calls 401/403; enforced at gateway guard, not client; attempts logged
- **Priority:** P0

### TC-SEC-006 — Webhook forgery and replay rejected
- **Type:** security
- **Requirement:** NFR-5, FR-10
- **Preconditions:** Webhook endpoint live; real HMAC secret
- **Steps:** 1. POST unsigned payload 2. POST valid-format wrong-signature 3. Replay a genuine webhook with timestamp outside window
- **Expected:** All three rejected (401/400) with alert on repeated forgery; genuine in-window duplicates remain safe no-ops (TC-INT-005)
- **Priority:** P0

### TC-SEC-012 — Partner users row-scoped to their own store
- **Type:** security
- **Requirement:** FR-29
- **Preconditions:** Partner users P1 (store S1), P2 (store S2)
- **Steps:** 1. As P1, request S2's orders, stock, settlement via UI and raw API (ID enumeration)
- **Expected:** All S2 access 403/404; no data leakage in list endpoints; attempts audit-logged
- **Priority:** P0
