## Parent story — E01-Th03-S01

**Title:** As a customer/rider, I want to log in with my phone and an OTP, so that I never manage a password

**User story:** As a customer/rider, I want to log in with my phone and an OTP, so that I never manage a password

**Acceptance criteria:**

- [ ] OTP delivered ≤ 30 s; login within ≤ 2 attempts; wrong/expired OTP always rejected
- [ ] Access JWT RS256 15 min + rotating hashed refresh (30 d) with reuse detection revoking the family
- [ ] Rate caps 3/min, 10/day per number plus per-IP caps, constant-time compare
- [ ] A rider token can never call customer or admin APIs

**Linked FRs:** FR-1
**Linked test cases:** TC-FUNC-001, TC-FUNC-002, TC-NEG-001, TC-SEC-001, TC-SEC-002, TC-SEC-003

## Test cases this task must make pass

### TC-FUNC-001 — Phone-OTP login succeeds
- **Type:** integration
- **Requirement:** FR-1
- **Preconditions:** SMS provider sandbox up; unregistered Indian mobile
- **Steps:** 1. Request OTP 2. Assert SMS dispatch ≤ 30 s 3. Submit correct OTP
- **Expected:** Account created/logged in; access JWT (15 min) + refresh token issued
- **Test data:** ``+919800000001`, fixture `auth/otp.json``
- **Priority:** P0

### TC-FUNC-002 — Session persists via refresh rotation
- **Type:** integration
- **Requirement:** FR-1
- **Preconditions:** Logged-in user; access token expired
- **Steps:** 1. Call API with expired JWT 2. Client refreshes 3. Retry call
- **Expected:** New access+refresh pair issued (old refresh invalidated); API call succeeds without re-login
- **Priority:** P0

### TC-NEG-001 — Wrong/expired OTP rejected within attempt budget
- **Type:** integration
- **Requirement:** FR-1
- **Preconditions:** OTP issued
- **Steps:** 1. Submit wrong OTP twice 2. Submit expired OTP
- **Expected:** Clear error each time; login still possible with correct OTP within ≤ 2 attempts per AC; expired OTP never accepted
- **Priority:** P0

### TC-SEC-001 — OTP brute-force and SMS-pump protections hold
- **Type:** security
- **Requirement:** NFR-5, FR-1
- **Preconditions:** Rate caps configured (3/min, 10/day per number; per-IP caps)
- **Steps:** 1. Script 10 OTP requests/min for one number 2. Rotate numbers from one IP 3. Brute-force 6-digit codes
- **Expected:** Per-number and per-IP caps enforced at edge + app; OTP compare constant-time; lockout/backoff after cap; alerts on pump pattern
- **Test data:** `attack scripts `sec/otp_abuse.ts``
- **Priority:** P0

### TC-SEC-002 — JWT expiry + refresh rotation + reuse detection
- **Type:** security
- **Requirement:** NFR-5
- **Preconditions:** Valid session
- **Steps:** 1. Use access token past 15 min 2. Rotate refresh 3. Replay the OLD refresh token
- **Expected:** Expired JWT rejected; rotation issues new pair; old-refresh reuse revokes the entire token family and forces re-login
- **Priority:** P0

### TC-SEC-003 — Audience claims isolate rider/customer/admin tokens
- **Type:** security
- **Requirement:** NFR-5, FR-29
- **Preconditions:** Valid rider JWT
- **Steps:** 1. Call customer order API and admin API with rider token 2. Vice versa with customer token
- **Expected:** All cross-audience calls 401/403; enforced at gateway guard, not client; attempts logged
- **Priority:** P0
