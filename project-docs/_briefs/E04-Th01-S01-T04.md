## Parent story — E04-Th01-S01

**Title:** As a picker, I want a layout-ordered, barcode-verified pick list with an OOS escape hatch, so that I pack fast and honestly

**User story:** As a picker, I want a layout-ordered, barcode-verified pick list with an OOS escape hatch, so that I pack fast and honestly

**Acceptance criteria:**

- [ ] `order.placed` puts the order in the store's picker queue sorted by promised ETA
- [ ] Pick list renders in store-layout order; scanning the wrong barcode can never confirm a line
- [ ] Pack complete is blocked until every line is picked / substituted / refunded; pack timestamps recorded
- [ ] OOS at pick sends a substitute-or-refund prompt with a 60 s timeout → auto-refund of that line

**Linked FRs:** FR-17, FR-30
**Linked test cases:** TC-FUNC-021, TC-FUNC-034, TC-NEG-021, TC-NEG-034, TC-E2E-005, TC-INT-003

## Test cases this task must make pass

### TC-FUNC-021 — Picker pick list in store-layout order with barcode verify
- **Type:** integration
- **Requirement:** FR-17
- **Preconditions:** New dark-store order, 4 SKUs with layout positions
- **Steps:** 1. Accept order 2. Scan each item 3. Complete pack
- **Expected:** List sorted by layout; wrong-item scan impossible to confirm; state PICKING→PACKED with pack timestamps recorded
- **Test data:** `fixture `stores/layout_a.json``
- **Priority:** P0

### TC-FUNC-034 — Substitution accepted → line substituted, bill adjusted
- **Type:** integration
- **Requirement:** FR-30
- **Preconditions:** Order in PICKING; SKU-2 OOS at shelf; substitute SKU-2b offered
- **Steps:** 1. Picker flags OOS 2. Customer accepts substitute within 60 s
- **Expected:** Line replaced with SKU-2b; price delta reflected in bill; order continues; customer notified
- **Priority:** P1

### TC-NEG-021 — Wrong barcode cannot confirm pick line
- **Type:** integration
- **Requirement:** FR-17
- **Preconditions:** Pick list active
- **Steps:** 1. Scan barcode of different SKU
- **Expected:** Line stays unconfirmed; mismatch feedback; pack-complete blocked until all lines resolved (picked/substituted/refunded)
- **Priority:** P0

### TC-NEG-034 — Customer rejects substitute → line refunded, order continues
- **Type:** integration
- **Requirement:** FR-30
- **Preconditions:** OOS flagged; substitute offered
- **Steps:** 1. Customer taps refund-instead 2. Complete order
- **Expected:** Line marked refunded, refund row created (BR-4 path), remaining lines delivered; bill reflects removal
- **Priority:** P1

### TC-E2E-005 — OOS at pick → substitution accepted → adjusted delivery
- **Type:** e2e
- **Requirement:** FR-30, FR-17
- **Preconditions:** Order in PICKING; one line OOS on shelf
- **Steps:** 1. Picker flags OOS 2. Customer accepts substitute within 60 s 3. Pack, dispatch, deliver
- **Expected:** Substituted line delivered; bill and invoice reflect substitution delta; second run with timeout → line auto-refunded (cross-check TC-NEG-034) and partial order delivered
- **Priority:** P1

### TC-INT-003 — Consumers are idempotent under duplicate delivery
- **Type:** integration
- **Requirement:** FR-28, Arch §3a C-9

- **Expected:** Exactly one notification, one audit row, one picker-queue entry; duplicates detected via processed-event key and skipped
