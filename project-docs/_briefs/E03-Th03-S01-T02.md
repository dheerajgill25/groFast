## Test cases this task must make pass

### TC-FUNC-013 — UPI payment success confirms order < 3 s
- **Type:** e2e
- **Requirement:** FR-10, NFR-2
- **Preconditions:** Gateway sandbox; funded test VPA
- **Steps:** 1. Checkout 2. Pay via UPI 3. Time capture-verify → confirmation
- **Expected:** Order confirmed < 3 s after gateway success; confirmation push received
- **Test data:** `sandbox VPA `success@upi``
- **Priority:** P0

### TC-FUNC-014 — Server-verified capture: verify-call and webhook both idempotent
- **Type:** integration
- **Requirement:** FR-10
- **Preconditions:** Sandbox payment captured
- **Steps:** 1. Deliver client verify callback 2. Deliver signed webhook for same `gateway_payment_ref` (any order)
- **Expected:** Whichever arrives first confirms; second is a no-op; exactly one order, one `payment.captured` event
- **Priority:** P0

### TC-NEG-011 — Payment failure → retry or switch method, cart intact
- **Type:** e2e
- **Requirement:** FR-10
- **Preconditions:** Sandbox failure VPA
- **Steps:** 1. Pay with failing method 2. Retry with card
- **Expected:** Clean failure message; cart and checkout session intact; second attempt succeeds; only one order results
- **Test data:** `sandbox VPA `failure@upi``
- **Priority:** P0

### TC-NEG-012 — Payment timeout leaves no phantom order
- **Type:** integration
- **Requirement:** FR-10
- **Preconditions:** Sandbox configured to never answer
- **Steps:** 1. Initiate payment 2. Abandon 3. Wait past 15 min hold
- **Expected:** No order created while PENDING; reconciliation sweep resolves (release or refund per gateway truth); customer messaged to retry
- **Priority:** P0

### TC-NEG-030 — Refund above ₹2,000 requires manual approval; ≤ ₹2,000 does not
- **Type:** integration
- **Requirement:** FR-26, BR-4
- **Preconditions:** Two cancelled prepaid orders: ₹1,999 and ₹2,001
- **Steps:** 1. Trigger both refunds
- **Expected:** ₹1,999 auto-initiates with no approval step; ₹2,001 queues for finance approval, then initiates; both audit-logged
- **Test data:** `amounts ₹1,999 / ₹2,001`
- **Priority:** P0

### TC-INT-005 — Duplicate payment webhook is a no-op
- **Type:** integration
- **Requirement:** FR-10, Arch §5e
- **Preconditions:** Captured payment; webhook already processed
- **Steps:** 1. Re-POST identical signed webhook 5×
- **Expected:** Upsert keyed on `gateway_payment_ref`+type: no state change, no duplicate order/refund/notification; 2xx returned each time
- **Priority:** P0

### TC-INT-006 — Out-of-order webhooks resolved by gateway timestamp
- **Type:** integration
- **Requirement:** FR-10, Arch §5e
- **Preconditions:** `payment.captured` (T2) processed
- **Steps:** 1. Deliver older `payment.failed` (T1) after
- **Expected:** Older event ignored by timestamp precedence; payment remains captured; raw payload still persisted for forensics
- **Priority:** P0

### TC-INT-007 — Lost webhook covered by reconciliation sweep
- **Type:** integration
- **Requirement:** FR-10, BR-3, Arch §5e
- **Preconditions:** Sandbox webhooks suppressed; payment captured at gateway
- **Steps:** 1. Wait past PENDING > 15 min 2. Sweep cycle (5 min) runs
- **Expected:** Sweep queries gateway status API, confirms capture, creates order (or refunds if reservation gone); BR-3 hold released on failed/expired cases
- **Priority:** P0

### TC-INT-008 — Captured-but-no-order auto-refunds immediately
- **Type:** integration
- **Requirement:** FR-10, BR-4
- **Preconditions:** Client killed between capture and reservation; reservation failed
- **Steps:** 1. Deliver capture webhook
- **Expected:** State comparison detects capture with no order → refund initiated in the same handling, idempotency key = refund ID; customer notified
- **Priority:** P0

### TC-INT-009 — Refund SLA monitor alerts before 24 h breach
- **Type:** integration
- **Requirement:** FR-14, BR-4
- **Preconditions:** Refund row with `initiated_at` NULL, created 20 h ago (clock-shifted)
- **Steps:** 1. Monitor cycle runs
- **Expected:** Ops alert fires at ≥ 20 h uninitiated; dashboard shows breach candidate; audit trail intact
- **Test data:** `clock-shift harness`
- **Priority:** P0

### TC-INT-018 — Gateway refund call idempotent on refund ID
- **Type:** integration
- **Requirement:** FR-14, BR-4
- **Preconditions:** Refund initiated; gateway call timed out ambiguously
- **Steps:** 1. Retry refund API call with same refund ID 3×
- **Expected:** Gateway receives idempotency key = refund ID; exactly one refund settles; refund row converges to single terminal state
- **Priority:** P0

### TC-SEC-005 — No raw card/UPI instrument data anywhere (PCI SAQ-A)
- **Type:** security
- **Requirement:** NFR-5, NFR-6
- **Preconditions:** Full E2E payment executed with tracing + debug logs on
- **Steps:** 1. Complete card + UPI sandbox payments 2. Sweep DB dumps, logs, traces, Redis, object storage for PAN/instrument patterns
- **Expected:** Zero raw instrument data at rest or in logs (regex sweep clean); only gateway tokens/refs stored; checkout served from gateway-hosted fields/SDK
- **Test data:** `PAN regex sweep `sec/pan_sweep.sh``
- **Priority:** P0

### TC-SEC-006 — Webhook forgery and replay rejected
- **Type:** security
- **Requirement:** NFR-5, FR-10
- **Preconditions:** Webhook endpoint live; real HMAC secret
- **Steps:** 1. POST unsigned payload 2. POST valid-format wrong-signature 3. Replay a genuine webhook with timestamp outside window
- **Expected:** All three rejected (401/400) with alert on repeated forgery; genuine in-window duplicates remain safe no-ops (TC-INT-005)
- **Priority:** P0

### TC-E2E-004 — Cancellation + refund lands within SLA on gateway sandbox
- **Type:** e2e
- **Requirement:** FR-14; BR-4
- **Preconditions:** Prepaid order in PACKED
- **Steps:** 1. Cancel 2. Track refund through gateway sandbox webhooks to terminal state
- **Expected:** Stock released instantly; refund initiated immediately (< 24 h SLA with 20 h alert never firing); refund webhook closes loop; customer notified at each step
- **Priority:** P0
