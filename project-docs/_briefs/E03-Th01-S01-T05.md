## Parent story — E03-Th01-S01

**Title:** As a customer, I want my order accepted only if the stock is really there and the charge is really made, so that I'm never disappointed or double-charged

**User story:** As a customer, I want my order accepted only if the stock is really there and the charge is really made, so that I'm never disappointed or double-charged

**Acceptance criteria:**

- [ ] Reservation and order creation are one Postgres transaction; any line shortfall rolls the whole thing back
- [ ] Under 50 concurrent runs on a single last unit, exactly one order wins and the loser gets a clean failure + auto-refund
- [ ] `eta_promised_at` is stored at placement and never mutates
- [ ] Checkout confirm is idempotent on the checkout-session ID; unpaid holds expire at 15 min

**Linked FRs:** FR-8, FR-12
**Linked test cases:** TC-FUNC-010, TC-FUNC-016, TC-NEG-015, TC-NEG-016, TC-NEG-036, TC-INT-001, TC-INT-014, TC-INT-017, TC-PERF-003

## Test cases this task must make pass

### TC-FUNC-010 — ETA promise displayed at checkout and stored on order
- **Type:** integration
- **Requirement:** FR-8, BR-2
- **Preconditions:** Store load + rider availability seeded in Redis
- **Steps:** 1. Reach checkout 2. Note ETA 3. Place order 4. Read order row
- **Expected:** Dynamic ETA shown (e.g., "12 min"); `eta_promised_at` on order equals the promise shown
- **Priority:** P0

### TC-FUNC-016 — Atomic reservation across all lines at payment confirmation
- **Type:** integration
- **Requirement:** FR-12, BR-3
- **Preconditions:** 3-line cart, all in stock
- **Steps:** 1. Complete payment 2. Inspect `store_inventory` + order in one read
- **Expected:** Single transaction: all lines `available`−/`reserved`+, order PLACED, outbox rows written; no intermediate state visible
- **Priority:** P0

### TC-NEG-015 — Stock race: two customers, one last unit
- **Type:** integration
- **Requirement:** FR-12
- **Preconditions:** SKU available_qty = 1; two paid checkouts fire concurrently
- **Steps:** 1. Trigger both confirmations in parallel (repeat ×50)
- **Expected:** Exactly one order wins the `FOR UPDATE` reservation every run; loser gets clean failure, cart intact, auto-refund initiated; never negative stock, never double-sell
- **Test data:** `harness `tests/race/last_unit.ts``
- **Priority:** P0

### TC-NEG-016 — Any line shortfall rolls back whole reservation
- **Type:** integration
- **Requirement:** FR-12
- **Preconditions:** 3-line cart; line 2 short by 1
- **Steps:** 1. Confirm payment
- **Expected:** Full rollback: no order row, no partial reservation, refund auto-initiated (BR-4), cart intact with shortfall reason
- **Priority:** P0

### TC-NEG-036 — Unpaid hold expires at 15 min and releases reservation
- **Type:** integration
- **Requirement:** BR-3
- **Preconditions:** Payment PENDING with reservation held (edge: reservation-before-capture path via gateway retry)
- **Steps:** 1. Hold reaches 15 min 2. Sweep runs
- **Expected:** Reservation released back to `available`; payment marked expired/failed per gateway truth; customer notified to retry
- **Priority:** P0

### TC-INT-001 — Outbox rows commit atomically with the state change
- **Type:** integration
- **Requirement:** FR-12, NFR-10, Arch §5a
- **Preconditions:** Checkout ready
- **Steps:** 1. Confirm payment (success run) 2. Force txn rollback on a second run (inject line shortfall)
- **Expected:** Success: order + `order.placed`/`payment.captured`/`stock.reserved` outbox rows in one commit. Rollback: zero outbox rows — an event can never exist without its state change
- **Priority:** P0

### TC-INT-014 — ETA promise immutable through lifecycle and drives breach math
- **Type:** integration
- **Requirement:** BR-2, FR-8
- **Preconditions:** Order placed with promise T+12 min
- **Steps:** 1. Drive full lifecycle incl. an ops ETA-param change mid-flight 2. Read order
- **Expected:** `eta_promised_at` never mutates after placement; exception-queue breach calc and SLA reports use the stored promise, not current params
- **Priority:** P0

### TC-INT-017 — Checkout confirm is idempotent on checkout-session ID
- **Type:** integration
- **Requirement:** FR-10, FR-12
- **Preconditions:** Payment captured for session S
- **Steps:** 1. POST confirm for S twice concurrently + once again after success
- **Expected:** One order, one reservation, one charge; retries return the same order ID (idempotency key = checkout-session ID at gateway and API layer)
- **Priority:** P0

### TC-PERF-003 — Order confirm p95 < 3 s at surge
- **Type:** load
- **Requirement:** NFR-2, FR-12
- **Preconditions:** Gateway sandbox; full checkout path
- **Steps:** 1. Sustain 7 orders/min (10k/day avg) 2. Surge ×5 (35/min) for 15 min
- **Expected:** Pay-confirm→confirmed p95 < 3 s throughout; zero lost/duplicate orders; outbox lag < 5 s
- **Test data:** ``perf/checkout.js``
- **Priority:** P0
